# Release Scope Lock

This file is the self-contained release-scope lock for the active EPaxos production-readiness goal.

## Scope rule

A release claim is allowed only when the item is listed in **Closed release items** with current evidence. If any item needed for the requested production-readiness goal is listed in **Open release items**, the release decision is no-go. New evidence must be added here before changing the decision.

## Current release decision

No-go.

The implemented EPaxos core, explicit TOQ core mode, deterministic timing subset, HTTP KV service-hardening changes, local fault campaign, operations artifacts, evidence bundle, and go/no-go workflow are ready for reviewer assessment under a simulation/local-loopback release scope. The broader production-readiness request is not complete because broader formal proof, target-environment deployment and data-lifecycle drills, real mixed-version upgrade/rollback execution, capacity measurements, and incident-readiness exercises remain open below. External multi-host Jepsen histories are outside the current release evidence.

## Non-claims

- Paper fast-quorum implementation parity is closed for odd supported sizes in Go with FP-deps-committed fast commits, sender-preserving committed stale-dependency evidence search/resend-ignore, targeted leader-in-fast-quorum/deferred-cycle handling, fail-closed slow accept, and finite `tla/EPaxosEvidenceQuery.tla` coverage of the read-only evidence-query path. Even sizes remain conservative by design, and unbounded proof plus complete optimized-recovery branch parity remain non-claims.
- Explicit TOQ core behavior assumes the embedding application supplies synchronized clock reads and one-way-delay estimates that include clock-skew/sync margin; `tla/TOQClockDiscipline.tla` checks only a finite bounded-skew/bounded-delay `ProcessAt` contract, and the core does not synchronize clocks, measure one-way delay, or prove an operational clock-discipline implementation.
- Local Jepsen passing and deterministic simulation do not claim target-environment external histories.
- The chaos campaign is a named local CI gate; it closes only simulation/local-loopback checkpoint-backed single-node corruption recovery. It does not close in-place Pebble/WAL repair, checksum recomputation, corrupt-record deletion, synthesized reconstruction without a verified checkpoint, production deployment proof, target capacity proof, or incident drill evidence listed as open.
- Request deadline budgets use explicit Go HTTP and handler deadlines; they are not OS-clock mutation evidence.
- Clock pause handling covers deterministic logical RawNode tick pause/resume, not OS scheduler suspension or wall-clock pause.
- VM rollback restart catch-up covers an in-memory RawNode/storage rollback and quorum catch-up without duplicate apply; it does not close in-place bit-level disk repair, synthesized reconstruction without a verified checkpoint, multi-replica/quorum-loss recovery, or actual mixed-version binary execution.
- API separation covers distinct client, peer-replication, and administrative HTTP listener/route planes; peer replication is POST-only and rejects other methods before body decode; TLS evidence is tracked separately under Transport security and does not add authentication or authorization.
- Transport security covers optional TLS listeners plus CA-configured peer HTTPS client verification; it does not implement client-certificate authentication, HTTP authentication, or authorization.
- Request size limits cover HTTP request body bytes on client, peer, and administrative planes; they do not bound scan result cardinality or keyspace traversal.
- Scan latest-read semantics are closed for the HTTP KV node's own write path by a shared scan-barrier conflict key; this does not claim protocol-native range/prefix conflict predicates for arbitrary EPaxos applications.
- Binary value fidelity covers raw point values plus `value_b64` for transaction/scan JSON; legacy `value` JSON fields remain text-compatible and are not a binary transport for invalid UTF-8.
- This lock records open work; it does not implement those open items.
