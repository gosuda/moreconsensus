# Model Correspondence and Scope Report

This report maps the current finite TLA+ models to the Go implementation, compares the current model suite against the attached original `EgalitarianPaxos` TLA+ module, and names implementation areas verified outside the models.

## Model suite scope

The repository now uses a split model suite rather than one monolithic implementation oracle:

| Model | Scope |
| --- | --- |
| `tla/EPaxos.tla` | Normal-case pre-accept, accept, commit, local conflict-derived attributes, quorum-gated fast/slow transitions, and deterministic minimal-component execution. |
| `tla/EPaxosResponses.tla` | Response quorum evidence for fast commit, slow accept, prepare outcomes, prepare-driven commit/accept transitions, and bounded prepare-recovery branch priority including `TryWitnessQuorum = FastQuorum + SlowQuorum - ReplicaCount`. |
| `tla/EPaxosOptimizedRecovery.tla` | Focused finite optimized-recovery model: prepare fast-witness threshold, recovery-only AcceptReply evidence that does not mutate chosen attributes, and an Accept-Deps stale-dependency slice where chosen `Deps` already order the conflict. The gate runs 3-, 5-, and 7-replica configurations. It abstracts the read-only message exchange into evidence availability; `tla/EPaxosEvidenceQuery.tla` covers the message path separately. |
| `tla/EPaxosTryPreAcceptBranches.tla` | Focused finite TryPreAccept abstract scenario/stage response-branch model: stale restart, committed evidence ignore/fail-closed, direct/forced accept, one uncommitted-conflict deferral with duplicate suppression, and OK slow-quorum accept. The only quorum detail modeled inside this slice is `okVotes >= SlowQuorum`; the gate runs 3-, 5-, and 7-replica configurations. This is not a TryPreAccept message-path model or unbounded recovery proof; concrete message paths and the implemented decision table are modeled separately. |
| `tla/EPaxosTryPreAcceptMessagePath.tla` | Focused finite TryPreAccept request/response message-path model for 3-, 5-, and 7-replica configurations: follower `MsgTryPreAccept` committed commit-only, stale reject, committed/uncommitted conflict reject reasons, duplicate matching PreAccepted re-ack without a durable rewrite, fresh PreAccepted durable ack, coordinator `MsgTryPreAcceptResp` stale restart, committed evidence/direct-accept branches, uncommitted forced/deferred branches, older-ballot ignore, duplicate OK ignore, first OK below quorum, pre-seeded witness/local OK immediate accept, and OK slow-quorum accept. It is not a full network, arbitrary evidence-query history, or unbounded recovery proof; the implemented decision table is modeled separately. |
| `tla/EPaxosTryConflictForce.tla` | Focused finite 3/5/7 model of `tryConflictForcesSlowAccept` / `leaderMustBeInCandidateFastQuorum`: a conflict leader forces slow Accept only when the candidate does not already depend on that conflict and every possible candidate fast quorum must include that leader; a deferred-cycle leader can independently force slow Accept when every possible candidate fast quorum must include that deferred leader; otherwise the uncommitted conflict is deferred and blocker recovery starts. It is not a full recovery or network model. |
| `tla/EPaxosEvidenceQuery.tla` | Focused finite committed-conflict TryPreAccept evidence-query model: guard-gated `MsgEvidence` start, read-only evidence responses, duplicate/mismatched response drops, sender-preserving `AcceptEvidence` validation, stale TryPreAccept rejection restart, and fail-closed slow-accept fallback. The gate runs 3-, 5-, and 7-replica configurations. |
| `tla/EPaxosEvidenceStaleness.tla` | Focused finite three-voter committed-conflict evidence staleness model: older-ballot `MsgEvidenceResp` is dropped before recording, same-ballot first response wins for duplicate senders, a stale/empty first response followed by a fresher committed duplicate from the same sender cannot authorize a TryPreAccept ignore resend, and all recorded empty remote responses fail closed to slow Accept. This is not an arbitrary network, retry, message-loss, or unbounded recovery proof. |
| `tla/EPaxosAcceptEvidenceMerge.tla` | Focused finite AcceptEvidence sender-merge and validation model: outbound zero-sender evidence is skipped, same-sender entries coalesce by `mergeAttrs` maxima into one tuple, distinct-sender evidence appends once, inbound validation permits identical duplicate sender evidence, and inbound validation rejects conflicting same-sender tuples. This is not an arbitrary evidence-history, network-delivery, recovery-branch, or unbounded proof. |
| `tla/EPaxosTryPreAcceptRetry.tla` | Focused finite TryPreAccept retry-timer model: current-ballot pending evidence for the same target preempts retry and fails closed to slow Accept; stale same-target evidence is deleted before retry; unrelated-target evidence is retained; and retry rebroadcasts carry either the normal TryPreAccept tuple or ignore markers without durable or application effects. This is not an arbitrary network, message-loss, recovery-under-reconfiguration, or unbounded recovery proof. |
| `tla/EPaxosOptimizedRecoveryDecisionTree.tla` | Focused finite optimized-recovery decision-table model for the implemented F<=3 Accept-Deps mode: CMU-PDL-13-111 Section 6.2 branches for no-information no-op, committed commit, accepted/preaccepted slow Accept, committed-conflict evidence authorize/fail-closed, committed-conflict direct slow Accept, uncommitted leader-required/deferred-cycle slow Accept, optional uncommitted defer-and-recover, stale restart, OK quorum slow Accept, and duplicate/old response ignore. It closes complete optimized-recovery branch parity for the implemented decision tree but does not model arbitrary networks, arbitrary message loss/retry schedules, recovery under reconfiguration, arbitrary durable histories, or unbounded recovery trees. |
| `tla/EPaxosRecovery.tla` | Focused stopped-owner dependency recovery: a non-owner coordinator recovers a missing dependency without owner votes, commits a no-op, and releases the dependent command under fairness. The gate runs 3- and 5-replica configurations. |
| `tla/EPaxosRollbackAllocation.tla` | Focused finite rollback-allocation model: a restored local checkpoint learns a later own committed instance from quorum state, advances `nextInstance`, exercises the defensive allocation skip over an already-known future local ref, allocates a fresh local instance, and checks that the learned commit appears before the fresh post-catch-up allocation in the apply sequence. It does not model full EPaxos recovery, message loss, storage checksums, application commands, or unbounded rollback histories. |
| `tla/EPaxosConfigBarrier.tla` | Focused finite configuration-barrier model: two fixed configuration-command refs plus one user command check local `pendingConf` blocking, dependency-vector inclusion, sequence elevation, and pending clearing after all known config refs execute. It does not model dynamic membership transitions, quorum changes, recovery under configuration changes, or unbounded instance spaces. |
| `tla/EPaxosConfigTransition.tla` | Focused finite add-voter dynamic configuration-transition model: one old three-voter configuration, one add-voter successor configuration, a config command chosen/executed under the old configuration, one old in-flight user instance that remains pinned to the old voters/quorum after the transition, and one later user instance pinned to the new voters/quorum. It does not model multi-step reconfiguration chains, joint consensus, recovery under configuration changes, durable replay, or unbounded configuration histories. |
| `tla/EPaxosConfigRemoveTransition.tla` | Focused finite remove-voter dynamic configuration-transition model: one old four-voter configuration, one remove-voter successor configuration, a config command chosen/executed under the old configuration, one old in-flight user instance that remains pinned to the old voters/quorum and can still count the removed voter after the transition, and one later user instance pinned to the new voters/quorum excluding the removed voter. It does not model joint consensus, concurrent configuration commands, recovery, message loss, durable replay, or unbounded configuration histories. |
| `tla/EPaxosConfigChainTransition.tla` | Focused finite add-then-remove configuration-chain model: an old three-voter configuration adds voter 4, then removes voter 2, while old and mid-flight user instances stay pinned to the voter/quorum domains they were created under and the final user instance uses only the final voter set. It does not model joint consensus, concurrent configuration commands, recovery, message loss, durable replay, arbitrary multi-step chains, or unbounded configuration histories. |
| `tla/EPaxosConfigChainTransitionRetry.tla` | Focused finite add-then-remove configuration-chain retry-timer model: after Conf1 `{1,2,3}` adds voter 4 and Conf2 `{1,2,3,4}` later removes voter 2, old Conf1 local-owner PreAccept/Accept retries target `{2,3}` with dependency width 3 while mid Conf2 retries target `{2,3,4}` with dependency width 4 including removed voter 2. It does not model arbitrary retry histories, message loss, joint consensus, arbitrary membership histories, durable replay, or unbounded proof. |
| `tla/EPaxosConfigChainTransitionLostResponseRetry.tla` | Focused finite add-then-remove configuration-chain transition lost-response retry model: after Conf1 `{1,2,3}` adds voter 4 and Conf2 `{1,2,3,4}` later removes voter 2, old Conf1 local-owner PreAccept/Accept slices reject/non-count current voter 4 before retry and advance only when lost removed voter 2 responds after retry, while mid Conf2 slices count current voter 4 below the old 3/4 quorum and advance only when lost removed voter 2 responds after retry; retry targets and dependency widths stay pinned to each record `Ref.Conf`. It does not model arbitrary message loss, arbitrary retry histories, joint consensus, arbitrary membership histories, durable replay, or unbounded proof. |
| `tla/EPaxosConfigChainRecovery.tla` | Focused finite mid-chain configuration-recovery model: after `{1,2,3}` adds voter 4 and then removes voter 2, an instance pinned to Conf2 `{1,2,3,4}` still requires the old 3/4 prepare/accept quorum; the current Conf3 quorum is insufficient in the intermediate prepare and accept states, added voter 4 and removed voter 2 both count only for the old pinned instance, and removed voter 2 cannot propose new current-config work. It does not model joint consensus, arbitrary membership histories, retries, message loss, or unbounded recovery. |
| `tla/EPaxosConfigChainRecoveryLostResponseRetry.tla` | Focused finite mid-chain configuration-recovery lost-response retry model: after `{1,2,3}` adds voter 4 and then removes voter 2, a Conf2 instance keeps deterministic retry rebroadcasts pinned to Conf2 peers `{2,3,4}` and dependency width 4; current Conf3 quorum `{1,4}` responses remain insufficient before deterministic retry, removed voter 2 prepare/accept responses are explicit pre-retry losses, replacement responses after retry complete the Conf2 3/4 quorum, and removed voter 2 cannot propose new current-config work. It does not model arbitrary message loss, arbitrary retry histories, arbitrary recovery histories, arbitrary membership histories, joint consensus, or unbounded proof. |
| `tla/EPaxosConfigTransitionRetry.tla` | Focused finite configuration-transition retry-timer model: local-owner PreAccept and Accept retry rebroadcasts for one old instance after voter removal and one old instance after voter addition target the old `Ref.Conf` peers, preserve old dependency width, include the removed old voter for removal, exclude the added current voter for addition, and produce no durable/application effects on pure retry. It does not model arbitrary retry histories, message loss, joint consensus, arbitrary membership histories, or unbounded proof. |
| `tla/EPaxosConfigTransitionLostResponseRetry.tla` | Focused finite normal configuration-transition lost-response retry model: local-owner PreAccept and Accept slices after voter removal and voter addition keep deterministic retry rebroadcasts pinned to the old `Ref.Conf` peers and dependency width; in the removal slices, local owner 1 plus surviving voter 2 is a current-config quorum but remains below the old 3/4 quorum until the removed voter 4 replacement response arrives after retry; in the addition slices, added voter 4 is rejected/ignored and the old voter 2 replacement response after retry completes the old 2/3 quorum. It does not model arbitrary message loss, arbitrary retry histories, arbitrary membership histories, joint consensus, or unbounded proof. |
| `tla/EPaxosConfigTransitionDedup.tla` | Focused finite configuration-transition response de-duplication model: remote PreAccept/Accept responses are modeled separately from the counted local owner vote; duplicate old-voter responses and added-current-voter responses do not satisfy old pinned quorums after removal or addition, while the modeled second distinct old remote response does. It does not model arbitrary message histories, joint consensus, arbitrary membership histories, or unbounded proof. |
| `tla/EPaxosConfigReplay.tla` | Focused finite durable configuration replay model: restart begins with the caller's initial config plus the last durable config state, replays executed config-change records in sorted instance order to reconstruct intermediate historical configs, keeps unexecuted config commands pending, and rejects local proposals from a removed voter. It does not model joint consensus, message loss, recovery under reconfiguration, arbitrary durable histories, or unbounded membership changes. |
| `tla/EPaxosConfigRecovery.tla` | Focused finite recovery-under-configuration-change model: one old four-voter instance remains pinned after the current configuration removes voter 4, staged prepare/accept recovery counts the removed voter only toward the old instance's old slow quorum, the smaller current three-voter quorum remains insufficient for that old instance, and the removed voter cannot propose new current-configuration work. It does not model joint consensus, arbitrary membership histories, message loss, arbitrary recovery under reconfiguration, or unbounded recovery proof. |
| `tla/EPaxosConfigRecoveryDedup.tla` | Focused finite lost+duplicate response de-duplication model for recovery under configuration removal: one old four-voter instance remains pinned after the current configuration removes voter 4, one lost response and one duplicate response leave prepare/accept quorums below the old 3/4 threshold, and recovery advances only after a distinct removed-voter response. It does not model joint consensus, arbitrary membership histories, message loss, arbitrary recovery under reconfiguration, or unbounded recovery proof. |

It does not model arbitrary recovery histories, message loss, joint consensus, arbitrary membership histories, or unbounded proof. |
| `tla/EPaxosConfigRecoveryLostResponseRetry.tla` | Focused finite configuration-recovery lost-response retry model: one old four-voter instance remains pinned after voter removal, one old-voter prepare response and one old-voter accept response are explicit finite pre-retry losses that do not count before deterministic retry rebroadcast, and replacement responses after retry complete recovery through the old 3/4 quorum. It does not model arbitrary message loss, arbitrary retry histories, arbitrary recovery histories, joint consensus, arbitrary membership histories, or unbounded proof. |
| `tla/EPaxosRevisited.tla` | Focused EPaxos Revisited-inspired implementation checks: explicit TOQ zero-sequence/empty-dependency envelopes, delayed originator assignment at `ProcessAt`, pending-TOQ decision blocking, deterministic receiver `ProcessAt` processing/fast-wait timing, and chain-pruning execution. | `tla/TOQClockDiscipline.tla` | Finite TOQ embedder clock-discipline contract: if receiver clocks are within configured skew and one-way delays stay within configured bounds, the sender's `ProcessAt = send-local-time + MaxDelay + SkewBound` is late enough for every sync-group member to have received the flagged PreAccept by its local `ProcessAt`. In Go/API terms, each configured `TOQOneWayDelay[id]` value must already include the receiver's clock-skew/synchronization uncertainty. This model does not implement or prove clock synchronization, one-way-delay measurement, or sync-group construction. |
| `tla/ReadyAdvance.tla` | Durable `Ready` / `Advance` prefix acknowledgement and retry obligations. |
| `tla/Quorum.tla` | Supported quorum tables and quorum intersections. |
| `tla/KVTimestampStaleness.tla`, `tla/KVOmissionRecovery.tla` | Example KV timestamp semantics and omission/recovery scenarios. |

TOQ operational-clock boundary: `tla/TOQClockDiscipline.tla` assumes the embedder has already supplied bounded clock skew, bounded one-way delay, and the intended sync group. The repository core consumes those values but does not implement NTP/PTP-style synchronization, delay measurement, drift monitoring, sync-group rollout procedure, or target-environment validation. Those items remain release-blocking operational-clock proof, not hidden model coverage.


## Paper-grounded claim map

| EPaxos fast quorum | EPaxos SOSP 2013 §4.3/§4.4, ACM proceedings pp.362-365, and CMU-PDL-13-111 §6 define the optimized fast quorum as `F + floor((F + 1) / 2)` including the leader for odd `N=2F+1`. This yields `N=3 -> 2`, `N=5 -> 3`, `N=7 -> 5`, and relies on matching recovery evidence. | `epaxos/quorum.go`, `tla/Quorum.tla`, `tla/EPaxos.tla`, and `tla/EPaxosResponses.tla` use the optimized paper threshold for odd supported sizes and keep the previous conservative threshold for even supported sizes (`N=4 -> 4`, `N=6 -> 5`) because the paper proof assumes odd `N`. | EPaxos Revisited chain pruning | NSDI 2021 §3, USENIX proceedings p.616, permits pruning a dependency already ordered through a higher-sequence transitive dependency. | Go and `tla/EPaxosRevisited.tla` model/check the implemented pruning rule. | Implemented claim. |

## Production completion gate

This gate defines the minimum evidence required before the project can be called production-complete under a mission-critical, high-assurance bar. The gate is intentionally blocker-driven: any failed blocker row keeps the release closed.

### Entry criteria

| Area | Entry criterion | Required evidence |

## Open correspondence risks

- The TLA+ model is finite-bounded for TLC, so it demonstrates correspondence for the configured state space rather than unbounded proof.
- `tla/EPaxos.tla` is a normal-case integrated safety model; prepare branch priority and TryPreAccept witness thresholds are checked through `tla/EPaxosResponses.tla`, and stopped-owner dependency recovery is checked through the focused `tla/EPaxosRecovery.tla` model rather than inlined into the normal-case model.
- `tla/EPaxosConfigBarrier.tla`, `tla/EPaxosConfigTransition.tla`, `tla/EPaxosConfigRemoveTransition.tla`, and `tla/EPaxosConfigChainTransition.tla` are focused finite configuration models. The barrier model checks `pendingConf`, dependency, sequence, and clearing obligations for two fixed config refs and one user command; the transition models check one add-voter successor, one remove-voter successor, and one add-then-remove chain. Arbitrary membership histories, arbitrary multi-step reconfiguration chains, joint consensus, recovery under config changes, durable storage/replay, and unbounded config histories remain outside these models.
- `tla/EPaxosRollbackAllocation.tla` is a finite local rollback-allocation bookkeeping model. It checks a single older checkpoint, one own committed instance learned from quorum, one defensive known-future local ref, fresh allocation after skip, and learned-before-fresh apply sequence only; it does not model full EPaxos recovery, storage checksums, message loss, application commands, arbitrary multi-replica rollback, or unbounded rollback histories.
- Optimized EPaxos fast quorum is implemented for odd supported cluster sizes with FP-deps-committed prefix evidence, durable RecordBallot recovery evidence is implemented in Go, and recovery-only Accept-Deps evidence is implemented for Accept/AcceptReply/Commit/PrepareReply plus sender-preserving read-only evidence queries. Go also implements targeted leader-in-fast-quorum/deferred-cycle TryPreAccept slices, committed-conflict slow-accept hardening, and committed stale-dependency evidence-search/resend-ignore for supported F<=3 configurations. `tla/EPaxosEvidenceQuery.tla` covers the finite read-only evidence-query path, `tla/EPaxosOptimizedRecoveryDecisionTree.tla` closes complete optimized-recovery branch parity for the implemented F<=3 Accept-Deps decision tree, even cluster sizes keep the conservative quorum, and unbounded proof remains outside the claim.
- `tla/EPaxosTryPreAcceptBranches.tla` adds an abstract scenario/stage TryPreAccept response-branch slice for 3-, 5-, and 7-replica configurations: stale restart, committed evidence ignore/fail-closed, direct/forced accept, one uncommitted deferral with duplicate suppression, and OK slow-quorum accept with `okVotes >= SlowQuorum` as the only quorum detail. It does not model the concrete TryPreAccept message path or unbounded recovery proof.

It still does not model an arbitrary network, arbitrary evidence-query histories, or unbounded recovery histories.

- `tla/EPaxosAcceptEvidenceMerge.tla` adds one finite sender-evidence merge/validation slice and generated `5/5` states: zero-sender outbound input leaves evidence unchanged, same-sender outbound entries merge by max attributes, distinct-sender evidence appends once, identical duplicate sender validation remains accepted, and conflicting duplicate sender validation rejects. Arbitrary evidence histories, message delivery, recovery branch choices, and unbounded proof remain open.
- `tla/EPaxosTryPreAcceptRetry.tla` adds one finite logical timer slice for TryPreAccept retry after pending-evidence checks: current-ballot same-target evidence fails closed before retry, stale same-target evidence is deleted, unrelated-target evidence is retained, and retry rebroadcast has no durable/application effects. It still does not model arbitrary network delivery, message loss, recovery under reconfiguration, or unbounded proof.
